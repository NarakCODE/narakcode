## Why this matters

Two small, independent fixes:

- **A. RSS feeds emit unescaped XML and crash on bad dates.** All three RSS
  routes interpolate `title`/`description` straight into XML, so any `&`, `<`, or
  `>` in a post or block title/description produces malformed XML that breaks the
  feed in readers. They also call `new Date(value).toISOString()` with no guard;
  a missing/invalid `createdAt` throws `RangeError: Invalid time value` and 500s
  the route (at build time, since these are `force-static`). Content is
  author-controlled today, so this is latent, but a single stray ampersand in a
  title silently breaks the feed.
- **B. The autogenerated `src/registry/transformed/` mirror is undocumented.**
  `AGENTS.md` lists some never-edit generated files but omits this directory, so a
  contributor (human or agent) can hand-edit it and have changes silently
  overwritten by the next `pnpm registry:build`.

## Steps

### Step A1: Add helpers to `src/utils/string.ts`

Add a minimal XML-escaper and a safe ISO-date helper:

```ts
const XML_ESCAPES: Record<string, string> = {
  "&": "&amp;",
  "<": "&lt;",
  ">": "&gt;",
  '"': "&quot;",
  "'": "&apos;",
}
export function escapeXml(value: string): string {
  return value.replace(/[&<>"']/g, (c) => XML_ESCAPES[c])
}

// Returns an RFC-safe ISO string, or null if the input isn't a valid date.
export function toISODateSafe(value: string | undefined | null): string | null {
  if (!value) return null
  const d = new Date(value)
  return Number.isNaN(d.getTime()) ? null : d.toISOString()
}
```

Order matters in `escapeXml`: `&` is in the character class with the others, and
the replacement is per-character, so `&` is escaped correctly without
double-escaping (each matched char maps once). Keep it as written.

**Verify**: `pnpm check-types` → exit 0.

### Step A2: Apply to all three RSS routes

In each route's `.map(...)` template:

- wrap title and description with `escapeXml(...)` (title, and the
  `description || ""`);
- replace `new Date(x).toISOString()` with `toISODateSafe(x)`, and skip items
  whose date is null (filter them out) OR omit the `<pubDate>` for those items.
  Prefer **filtering out** items with an invalid date in `blocks/rss` (where
  `createdAt` can be undefined) so the feed never contains an item without a date;
  for `blog`/`components`, posts are expected to have dates, but use the same safe
  helper so a bad value can't 500 the build.

Target shape (blog example):

```ts
import { escapeXml, toISODateSafe } from "@/utils/string"

// ...
const itemsXml = getAllDocs()
  .map((doc) => {
    const pubDate = toISODateSafe(doc.metadata.createdAt)
    if (!pubDate) return null
    return `<item>
        <title>${escapeXml(doc.metadata.title)}</title>
        <link>${SITE_INFO.url}/blog/${doc.slug}</link>
        <description>${escapeXml(doc.metadata.description || "")}</description>
        <pubDate>${pubDate}</pubDate>
      </item>`
  })
  .filter(Boolean)
  .join("\n")
```

Apply the analogous change to `components/rss` (uses `getDocsByCategory("components")`)
and `blocks/rss` (uses `block.name`/`block.description`/`block.createdAt`).

**Verify**: `pnpm check-types` → exit 0; `pnpm lint` → exit 0.
