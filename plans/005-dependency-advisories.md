## Why this matters

`pnpm audit --prod` reports 32 advisories (2 critical, 15 high, 15 moderate).
Vetting shows they are **not** a directly-reachable attack surface on this static
portfolio — they are transitive, and the dominant cluster comes from one
dependency: `@c15t/nextjs` (the cookie-consent manager), which pulls in a full
server SDK (`@orpc/*`, `@opentelemetry/*`, `@grpc/grpc-js`, `protobufjs`,
`drizzle-orm`, `kysely`, `dompurify`, `js-cookie`). Database drivers and an
OpenTelemetry/gRPC stack are not used at runtime by a consent banner on this
site. The remaining advisories are build-time only (`ts-morph` → `minimatch` →
`brace-expansion`, `yaml`, `postcss` via `fumadocs-typescript`). So the honest
framing is dependency hygiene, not an emergency: bring the noisy tree up to
patched versions (or prune what isn't needed) so the audit is meaningful and the
2 criticals stop showing up.

This plan also removes one genuinely unused production dependency (`react-use`).

## Steps

### Step 1: Remove the unused dependency

```bash
pnpm remove react-use
```

**Verify**: `grep -rn "react-use" src` → no matches; `pnpm check-types` → exit 0;
`pnpm build` → exit 0.

### Step 2: Try the low-effort, low-risk update first

Check whether a newer `@c15t/nextjs` within or above the current range clears the
critical/high cluster:

```bash
pnpm outdated @c15t/nextjs
pnpm up @c15t/nextjs@latest
pnpm audit --prod
```

Compare the new advisory counts to the Current state numbers.

**Verify**: `pnpm build` exits 0 AND `pnpm dev` shows the consent banner still
appears and functions (open in a fresh browser profile / clear the consent
cookie). If the banner behaviour changed, STOP (see conditions).

### Step 3: If updating doesn't clear them, pin patched transitive versions with overrides

If `@c15t/nextjs@latest` still surfaces criticals from `protobufjs`/`@orpc/client`,
and the advisory pages list a patched version, add targeted `pnpm.overrides` to
`package.json` to force the patched transitive versions. Only add overrides for
packages the audit flags as having a fixed version available. Example shape (use
the actual patched versions from the advisory output — do not copy these):

```json
"pnpm": {
  "overrides": {
    "protobufjs": ">=<patched>",
    "@orpc/client": ">=<patched>"
  }
}
```

Then:

```bash
pnpm install
pnpm audit --prod
pnpm build
```

**Verify**: criticals are gone or reduced; `pnpm build` exits 0; consent banner
still works in `pnpm dev`.

### Step 4: Record what remains

Whatever advisories remain (e.g. build-time `brace-expansion` with no clean
upstream fix), note them in the PR description with one line each on why they are
not addressed (transitive, build-only, no patched version, or would force a
breaking app change). Do not silently leave them unexplained.

**Verify**: `pnpm registry:validate` → exit 0; `pnpm lint` → exit 0.

  the grep missed a usage; report it.
